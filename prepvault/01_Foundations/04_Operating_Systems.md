# Operating Systems (OS) Deep Dive

## Overview
An Operating System (OS) is the intermediary between computer users and computer hardware. It provides an environment in which a user can execute programs in a convenient and efficient manner. For software engineers, understanding the OS is crucial for writing performant, concurrent, and resource-efficient code.

---

## 1. Processes and Threads

### The Process
A **Process** is a program in execution. It is the unit of work in a modern time-sharing system.
- **Process State**:
    - *New*: The process is being created.
    - *Running*: Instructions are being executed.
    - *Waiting*: The process is waiting for some event (such as I/O completion).
    - *Ready*: The process is waiting to be assigned to a processor.
    - *Terminated*: The process has finished execution.
- **Process Control Block (PCB)**: A data structure containing information about the process (ID, state, program counter, registers, memory limits, list of open files).

### The Thread
A **Thread** is a basic unit of CPU utilization, comprising a thread ID, a program counter, a register set, and a stack.
- **Multithreading**: Threads within the same process share the code section, data section, and OS resources (like open files).
- **Benefits**: Responsiveness, resource sharing, economy (context switching threads is cheaper than processes), utilization of multiprocessor architectures.
- **User vs. Kernel Threads**: User threads are managed by a thread library; kernel threads are managed directly by the OS.

---

## 2. Process Scheduling
The objective of multi-programming is to have some process running at all times to maximize CPU utilization.

### Scheduling Algorithms
- **First-Come, First-Served (FCFS)**: Simple but suffers from the *convoy effect* (short processes waiting for one long process).
- **Shortest-Job-First (SJF)**: Gives the minimum average waiting time. Can be preemptive (Shortest-Remaining-Time-First).
- **Round Robin (RR)**: Designed for time-sharing systems. Each process gets a small unit of CPU time (time quantum).
- **Priority Scheduling**: CPU is allocated to the process with the highest priority. Starvation is a risk (low priority processes never run); solved by *aging*.
- **Multilevel Queue**: Partition the ready queue into separate queues (e.g., foreground and background) with different scheduling needs.

## 3. Process Synchronization
In a multiprocessor environment, concurrent access to shared data may result in data inconsistency.

### Critical Section Problem
A segment of code where shared resources are accessed.
- **Requirements**: Mutual Exclusion, Progress, Bounded Waiting.
- **Tools**:
    - **Mutex**: A simple lock mechanism. A process must acquire the lock before entering the critical section.
    - **Semaphores**: More robust than mutexes. *Counting Semaphores* can control access to a resource with multiple instances.
    - **Monitors**: A high-level abstraction that provides a convenient and effective mechanism for process synchronization.

### Classic Problems of Synchronization
- **Producer-Consumer**: Coordinating producers and consumers sharing a fixed-size buffer.
- **Readers-Writers**: Multiple readers can read simultaneously, but only one writer can write at a time.
- **Dining Philosophers**: A classic concurrency problem illustrating deadlock and starvation.

---

## 4. Deadlocks
A situation where a set of processes are blocked because each process is holding a resource and waiting for another resource held by another process.

### Necessary Conditions
Deadlock can arise if these four conditions hold simultaneously:
1. **Mutual Exclusion**: At least one resource must be non-sharable.
2. **Hold and Wait**: A process must be holding at least one resource and waiting for others.
3. **No Preemption**: Resources cannot be taken away; they must be released voluntarily.
4. **Circular Wait**: A set of processes `{P0, P1, ..., Pn}` such that `P0` waits for `P1`, `P1` waits for `P2`, ..., and `Pn` waits for `P0`.

### Handling Deadlocks
- **Prevention**: Ensuring at least one of the four conditions cannot hold.
- **Avoidance**: Using knowledge about future resource requests (e.g., **Banker's Algorithm**).
- **Detection and Recovery**: Letting deadlocks happen, detecting them, and then terminating processes or preempting resources.

## 5. Memory Management

### Main Memory
- **Logical vs. Physical Address**: The address generated by the CPU is a *logical address*; the address seen by the memory unit is a *physical address*. The **Memory Management Unit (MMU)** maps logical to physical.
- **Swapping**: A process can be swapped temporarily out of memory to a backing store (disk).
- **Paging**: A memory management scheme that eliminates the need for contiguous allocation of physical memory. Memory is broken into fixed-size blocks called *frames*, and logical memory into *pages*.
- **Segmentation**: Memory management that supports the user's view of memory (code, stack, variables).

### Virtual Memory
- **Demand Paging**: Bringing a page into memory only when it is needed.
- **Page Replacement Algorithms**:
    - *FIFO*: First-in, first-out.
    - *Optimal*: Replace the page that will not be used for the longest period of time (impossible to implement, used as a benchmark).
    - *LRU (Least Recently Used)*: Replace the page that has not been used for the longest period of time.
- **Thrashing**: A high frequency of paging where the CPU spends more time swapping than executing.

---

## 6. Storage and File Systems
- **Disk Scheduling**: Algorithms like FCFS, SSTF (Shortest Seek Time First), SCAN (elevator), and C-SCAN to optimize disk I/O.
- **File Allocation Methods**:
    - *Contiguous*: Fast but suffers from external fragmentation.
    - *Linked*: No fragmentation, but slow for random access.
    - *Indexed*: (e.g., Inodes in Linux) Brings all pointers together into one location.

## 7. Advanced Operating System Concepts

### Linux Specifics: The VFS and Inodes
In Linux, "everything is a file." This is managed through the **Virtual File System (VFS)**.
- **Inodes**: A data structure on a filesystem that describes a file-system object such as a file or a directory. It contains metadata (owner, permissions, size, timestamps) and pointers to the actual data blocks on disk.
- **Hard Links vs. Symbolic Links**:
    - *Hard Link*: Multiple filenames pointing to the same inode. Deleting one doesn't delete the data as long as another link exists.
    - *Symbolic Link (Symlink)*: A special file that contains a path to another file. If the original file is deleted, the symlink breaks.

### Virtual Memory: TLB and Inverted Page Tables
- **Translation Lookaside Buffer (TLB)**: A hardware cache that stores recent translations from virtual memory to physical memory. This significantly speeds up address translation.
- **TLB Miss**: If an address is not in the TLB, the CPU must walk the page table (slow), then update the TLB.
- **Inverted Page Tables**: Instead of one page table per process, there is one page table for the entire system, with one entry for each real page (frame) in memory. This saves significant memory in systems with many processes.

### System Calls
A system call is the programmatic way in which a computer program requests a service from the kernel of the operating system it is executed on.
- **Examples**: `fork()` (create process), `exec()` (execute program), `wait()` (wait for process), `open()`, `read()`, `write()`, `close()`.
