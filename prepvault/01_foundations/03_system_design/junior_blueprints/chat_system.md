# System Design Blueprint: Real-Time Chat System (e.g., WhatsApp/Slack)

Focuses on bi-directional communication, state management, and persistence.

---

## 1. Requirements
- **Functional**:
    - One-on-one and group messaging.
    - User online/offline status (Presence).
    - Message history/persistence.
- **Non-Functional**:
    - Low latency (Real-time experience).
    - Consistency (Order of messages must be preserved).
    - High availability.

## 2. High-Level Architecture
- **WebSockets**: For bi-directional, persistent connection between client and server.
- **API Gateway**: For authentication, rate limiting, and HTTP requests (e.g., login, profile update).
- **Chat Service**: Orchestrates message delivery.
- **Presence Service**: Tracks which users are currently online (often uses Heartbeats).

## 3. Communication Protocol
- **WebSockets**: Standard for real-time.
- **XMPP / MQTT**: Older but robust protocols for messaging.
- **Long Polling**: Fallback if WebSockets are not supported.

## 4. Storage Strategy
- **Relational DB**: For User profiles, group metadata.
- **NoSQL (Key-Value/Columnar)**: For message history (e.g., **Cassandra** or **HBase**). 
    - *Why?* Chat messages are write-heavy and usually accessed sequentially by `channel_id + timestamp`.
- **In-Memory (Redis)**: For Presence status and transient data.

## 5. Key Challenges
- **Message Ordering**: Use a sequence number or a precise timestamp generated by the server.
- **Scalability**: How to handle 1M concurrent WebSocket connections? (Load balancing at the connection level, sharding chat rooms).
- **Offline Messages**: If a user is offline, store the message in the DB and push a notification (FCM/APNS).

---

## Role-Specific Applications
- **Frontend**: Managing WebSocket state, handling message optimistic updates, virtualized chat lists for performance.
- **Backend**: Handling massive concurrency, ensuring ACID for group metadata, implementing "read receipts".
- **DevOps**: Managing persistent connections (sticky sessions), scaling WebSocket clusters.
